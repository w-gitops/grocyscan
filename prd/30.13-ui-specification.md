# 30.13 UI/UX Specification

## 30.13.0 Design Philosophy (2026)

- **Tablet-first**: Design at tablet viewport (768â€“1024px) first, then adapt down to mobile (320â€“767px) and up to desktop (1025px+). Primary use case is scanning and inventory on iPad/tablet in kitchen or warehouse.
- **Mobile-optimized**: All flows must work one-handed on phone (360px minimum). Touch targets, navigation, and forms must meet mobile usability standards.
- **Desktop-compatible**: Full layout and keyboard/mouse support on large screens; no touch-only interactions that block desktop users.
- **PWA**: Installable, offline-capable web app with app-like experience when added to home screen (iOS "Add to Home Screen", Android install prompt where supported).
- **Apple alignment**: Follow [Apple Human Interface Guidelines](https://developer.apple.com/design/human-interface-guidelines/) where applicable for web apps on iPhone and iPad: clear hierarchy, predictable gestures, safe areas, and native-feeling controls.

## 30.13.1 UI Architecture

The UI is built using Vue 3 with Quasar Framework v2. Quasar provides Material Design components with responsive modes and built-in PWA support. Use **Vite** with **vite-plugin-pwa** for service worker and web app manifest generation (recommended over Vue CLI in 2026).

### Technology Stack

| Component | Technology | Purpose |
|-----------|------------|---------|
| Framework | Vue 3 + Composition API | Reactive UI |
| Component Library | Quasar v2 | Material Design |
| State Management | Pinia | Global state |
| Routing | Vue Router | SPA navigation |
| Build | Vite | Fast dev/build |
| Tables | QTable + AG Grid | Data grids |

> **Current Implementation Note:** The codebase currently uses **NiceGUI** (Python, server-rendered via WebSocket) served by FastAPI on port 3334 as an interim solution for rapid prototyping. The Vue 3 + Quasar stack described in this document is the **target architecture** for Phase 3. The NiceGUI UI provides functional pages but does not yet include PWA features (service worker, manifest, offline support). Responsive design principles (safe areas, touch targets, breakpoints) apply to both the current and target implementations. See [DEC-051](91.14-decision-log.md#dec-051-ui-stack-and-pwa-alignment) for details.

### 30.13.1.1 Apple / iOS PWA Requirements

For reliable behavior and usability when the app is used on iPhone and iPad (including "Add to Home Screen"):

- **Viewport and safe areas**
  - Use `viewport-fit=cover` so content can extend edge-to-edge; then respect safe areas so content is not hidden by notch, Dynamic Island, or home indicator.
  - In CSS, use `env(safe-area-inset-top)`, `env(safe-area-inset-bottom)`, `env(safe-area-inset-left)`, `env(safe-area-inset-right)` for padding/margins (e.g. bottom navigation must include `padding-bottom: env(safe-area-inset-bottom)`).
- **Meta tags** (in `index.html` or equivalent):
  - `<meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">`
  - `<meta name="apple-mobile-web-app-capable" content="yes">`
  - `<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">` (or `default` / `black` as desired)
- **Web app manifest**
  - `display: "standalone"` for app-window experience when installed.
  - `theme_color` (and optional `background_color`) for status bar / splash; ensure contrast with notch area.
  - **Icons** (required sizes):
    - 180Ã—180 PNG: Apple home screen icon.
    - 192Ã—192 PNG: Android standard icon.
    - 512Ã—512 PNG with `purpose: "maskable"`: Android adaptive icons (safe zone design requiredâ€”keep logo within center 80%).
    - 512Ã—512 PNG with `purpose: "any"`: Fallback for contexts that don't support masking.
  - **Splash screen**: Set `background_color` and `theme_color` in manifest; provide appropriately sized icons so the splash screen looks correct during app launch on iOS and Android.
- **iOS limitations to document for product/UX**: No install banner; users add via Share â†’ "Add to Home Screen". Push notifications and some PWA APIs are limited or unsupported on iOS; design for optional/graceful degradation.

### 30.13.1.2 Offline-First

The app should remain usable on poor or no connectivity. Avoid blocking the entire UI with "You are offline"; surface connectivity status and allow core flows (e.g. viewing cached data, queuing actions) where possible.

- **Service worker**: Use **vite-plugin-pwa** (Workbox under the hood) to generate and register a service worker. Precache the shell (HTML, JS, CSS, app icons) so the app loads from cache when offline.
- **Cache strategy**:
  - **Shell / static assets**: Cache-first (precache at build or install). Version via build hash so updates invalidate old caches.
  - **API responses**: Network-first with cache fallback for read-only data (e.g. product list, locations) where staleness is acceptable; use short TTL or versioned cache keys. For mutations (add stock, consume), do not rely on cacheâ€”fail gracefully with a clear "Offlineâ€”try again when connected" message and optional retry/queue.
- **Offline UX**:
  - Show a small, non-blocking indicator when offline (e.g. banner or icon). Do not prevent navigation or viewing cached content.
  - For scan/add flows that require the backend: show a clear message and retry option.
- **Offline action queue** (required):
  - When offline, allow core mutations (add stock, consume, transfer) to be **queued locally** (IndexedDB or localStorage).
  - Display queued actions with a "pending sync" indicator.
  - When connectivity is restored, **automatically sync** queued actions to the backend in order.
  - Handle sync conflicts gracefully (e.g., product no longer exists, stock already consumed)â€”notify user of any failed syncs.
- **Testing**: Verify in DevTools â†’ Application â†’ Service Workers (bypass for network, offline), and on a real device with airplane mode or weak signal.

### 30.13.1.4 Performance & Core Web Vitals

The app must meet Core Web Vitals thresholds for a responsive, professional user experience. Measure using Lighthouse (or PageSpeed Insights) and target the "Good" range.

| Metric | Target | Description |
|--------|--------|-------------|
| **LCP** (Largest Contentful Paint) | â‰¤2.5s | Main content visible within 2.5 seconds |
| **INP** (Interaction to Next Paint) | â‰¤200ms | Responsive to user input |
| **CLS** (Cumulative Layout Shift) | â‰¤0.1 | Minimal unexpected layout shifts |
| **FCP** (First Contentful Paint) | â‰¤1.8s | First content renders quickly |
| **TTI** (Time to Interactive) | â‰¤3.5s | App fully interactive |

**Performance guidelines:**

- **Avoid blocking the main thread** during scan/lookup operations; use async/background processing.
- **Lazy-load** heavy components (e.g., AG Grid, label designer) to improve initial load.
- **Optimize images**: Use WebP/AVIF formats, appropriate sizes, and lazy loading for product images.
- **Bundle splitting**: Vite/Rollup should split vendor chunks; avoid loading all JS upfront.
- **Lighthouse CI** (optional): Run Lighthouse in CI pipeline to catch performance regressions.

**User experience impact:** Load time directly affects engagementâ€”a 1-second load is optimal; as load time increases to 10 seconds, bounce probability increases significantly.

### 30.13.1.5 Desktop PWA (Optional)

For installed PWAs on desktop (Windows, macOS, ChromeOS), consider these optional enhancements:

- **display_override** in manifest: Use `["window-controls-overlay", "standalone"]` to enable a customizable title bar area when supported (Chrome 93+).
- **Window controls overlay**: If enabled, the app can render content in the title bar area (e.g., app name, quick actions) while native window buttons remain visible.
- **File handling** (future): PWAs can register as file handlers for specific types (e.g., `.homebot` export files).

These are **[FLEX]** features and not required for MVP.

## 30.13.3 Component Standards

### 30.13.3.1 Quasar Dialog (Standard Width)

Use a **centered card** on tablet/desktop; on **small viewports (xs, optionally sm)** use either a **full-width / maximized** dialog or Quasar's **Bottom Sheet** plugin so content sits above the keyboard and feels native on iPhone/iPad. Ensure all dialogs respect safe areas (padding using `env(safe-area-inset-*)`).

```vue
<template>
  <q-dialog v-model="show" persistent :maximized="$q.screen.lt.sm" :full-width="$q.screen.lt.sm">
    <q-card :style="dialogStyle" class="rounded-borders">
      <q-card-section class="row items-center q-pa-md" style="padding-top: max(16px, env(safe-area-inset-top))">
        <div class="text-h6">{{ title }}</div>
        <q-space />
        <q-btn icon="close" flat round dense v-close-popup size="lg" />
      </q-card-section>

      <q-separator />

      <q-card-section class="scroll" style="max-height: 70vh">
        <slot />
      </q-card-section>

      <q-separator />

      <q-card-actions align="right" class="q-pa-md" style="padding-bottom: max(16px, env(safe-area-inset-bottom))">
        <q-btn flat label="Cancel" v-close-popup size="lg" />
        <q-btn color="primary" label="Confirm" @click="onConfirm" size="lg" />
      </q-card-actions>
    </q-card>
  </q-dialog>
</template>
```

- **dialogStyle**: On `lt.sm`, use full width and max-height (e.g. `max-height: 90vh`); on `sm+`, use `width: 700px; max-width: 90vw`. Avoid fixed 700px width on narrow screens.

### 30.13.3.2 Touch-Friendly Date Picker

Use **quick-select buttons** (TODAY, +3D, +1W, etc.) as the primary touch path; they meet Apple's 44pt target and reduce reliance on popup. On iOS, `q-date` in a popup is acceptable; for maximum native feel, consider `type="date"` input (triggers system picker on iOS) as an option for simple "single date" fields.

      </template>

## 30.13.5 Touch Optimization & Input Methods

### Input Method Support

Support **all input methods** so no user is blocked regardless of device or preference:

- **Touch**: Primary for tablet and mobile; all interactions must work via tap/swipe.
- **Mouse**: Full support on desktop; hover states are optional enhancements only.
- **Keyboard**: All interactive elements focusable and operable via keyboard; support Tab navigation, Enter/Space activation, Escape to close dialogs.
- **Stylus / Apple Pencil**: Touch interactions must work with stylus; precision tasks (e.g., label designer) should benefit from stylus input.

No critical workflow may be **touch-only** or **desktop-only**; all primary actions must be accessible across all input methods.

### Touch Targets

- **Minimum tap target**: At least **44Ã—44 pt** (Apple HIG); **48Ã—48 px** (Quasar default) meets and exceeds this. Do not use smaller interactive hit areas.
- **Button padding**: Use `size="lg"` or equivalent for primary touch actions; ensure icon-only buttons have 44px+ tappable area (padding around icon counts).
- **Spacing**: Use `q-gutter-md` between interactive elements to avoid accidental taps.

### Gestures

- **Swipe gestures**: Use Quasar's `v-touch-swipe` where it enhances flow (e.g. dismiss sheet); avoid swipe as the only way to complete a critical action.
- **No hover-only actions**: All primary actions must be available via tap; hover can add secondary behavior on desktop only.

### One-Handed / Thumb Zone (Mobile)

On mobile (xs/sm), design for **one-handed use**:

- **Primary actions** (scan, add, consume) should be in the **thumb zone**â€”bottom half of the screen or bottom navigation bar.
- **Secondary actions** (settings, filters, search) may be at the top.
- Avoid requiring users to reach the top corners for frequently-used actions.

## 30.13.6 Accessibility

### Semantic HTML

Use **semantic HTML elements** instead of generic divs with custom roles:

- Use `<button>` for buttons, `<a>` for links, `<input>` for form fields.
- Use `<nav>`, `<main>`, `<header>`, `<footer>`, `<section>`, `<article>` for page structure.
- Use `<table>`, `<th>`, `<td>` for tabular data (not CSS grids for data tables).
- Add ARIA attributes only when semantic HTML is insufficient (e.g., `aria-label` for icon-only buttons, `aria-expanded` for custom dropdowns).

Semantic HTML ensures compatibility with all input methods, screen readers, and assistive technologies without extra effort.

### Screen Readers & Assistive Technology

- **VoiceOver / screen readers**: Ensure labels, headings, and button names are announced correctly. Test critical flows (scan, add product, change location) with VoiceOver on iOS and TalkBack on Android.
- **Heading hierarchy**: Use proper heading levels (`h1` through `h6`) in order; one `h1` per page.
- **Form labels**: All form inputs must have associated `<label>` elements or `aria-label`.

### Visual Accessibility

- **Dynamic Type**: Prefer relative units (rem, em) or scale-friendly typography so that system font-size changes (Settings â†’ Display â†’ Text Size) do not break layout. Avoid fixed small font sizes for essential copy.
- **Color and contrast**: Meet WCAG 2.1 Level AA for text and UI components. Do not rely on color alone to convey status (e.g. low stock: icon or text in addition to color).
- **Reduced motion**: Respect `prefers-reduced-motion: reduce` for non-essential animations (e.g. disable or shorten transitions). Quasar and CSS media queries can gate motion.

### Keyboard Navigation

- **Focus management**: On desktop, all interactive elements must be focusable and operable via keyboard; visible focus indicators must not be removed.
- **Focus trapping**: Dialogs and modals should trap focus while open and return focus to the trigger element on close.
- **Skip links**: Consider adding "Skip to main content" link for keyboard users.
